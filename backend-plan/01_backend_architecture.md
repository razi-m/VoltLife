# 01 — Backend Architecture (BINDING for Antigravity)

One FastAPI process, one Postgres, one WebSocket channel. The backend is the **orchestrator and system of record**; it never predicts and never recalculates ML outputs.

## Ownership boundaries

| Backend owns | ML subsystem owns (ml-plan/) |
|---|---|
| Intake, validation, normalization | SoH, RUL (+ intervals, cycles) |
| Lifecycle pipeline orchestration | Confidence tiers |
| Deployment *decisions* (wrapping the pure scoring engine) | Explainability (SHAP → reasons) |
| Aadhaar/passport generation + ledger | Grades (rules over predictions) |
| Impact accounting | Pure recommendation scoring (`ml/recommend.py`) |
| WS events, jobs, demo machinery, persistence | — |

The seam: backend imports `ml.predictor.assess()` and `ml.recommend.recommend()` as vendored pure functions (in-process — no ML microservice, per docs/02). Backend treats their outputs as **opaque facts**: stored, routed, broadcast — never modified beyond serialization.

## Data flow (= the mandated architecture, mapped to modules)

```
POST /batteries/ingest ──→ services/ingestion (validate, normalize, persist, reject report)
        │ returns job_id; spawns BackgroundTask
        ▼
services/pipeline  (per battery, sequential, paced):
  1. features.from_telemetry(summary)            ← shared ml/features.py
  2. ml.predictor.assess(features)               ← SoH/RUL/confidence/grade/reasons (NEVER recalculated)
  3. services/deployment.decide(...)             ← gates + ml.recommend + capacity decrement + persist
  4. services/aadhaar.issue(...)                 ← BPAN-style ID, ledger events, life_story
  5. services/impact (per-battery math, doc 06 formulas)
  6. core/events.broadcast(assessment / deployment / impact)   + asyncio.sleep(PACE_S)
        ▼
Dashboard reads: REST for state, WS (or polling fallback) for motion
```

## Cross-cutting decisions

- **Auth = "simple demo authentication"** as already contracted: no user auth; `X-DEMO-KEY` header guards destructive routes (`/demo/reset`, `/demo/replay`). Nothing else — auth has zero judge value (docs/02).
- **API versioning:** everything under `/api/v1/` (already frozen in docs/04). `/healthz` and `/ws/feed` sit outside the version prefix.
- **Alembic (mandated stack):** used as a **baseline-only** migration — one autogenerated initial revision, `alembic upgrade head` at startup. No incremental migrations during the event; the schema is frozen (docs/03). This honors the approved stack without the mid-hackathon-migration failure mode docs/08 warned about (docs/08 synced).
- **QR generation:** backend owns the QR *payload* (`qr_payload` URL in the aadhaar response — frozen contract); pixel rendering stays client-side (`qrcode.react`, docs/07 frozen). Print artifacts (hero-battery card) come from an offline script `scripts/make_qr_card.py` — not an endpoint. This is the one place the mandate ("QR Generation: Backend") is satisfied in spirit rather than letter, because adding a `/qr.png` endpoint would modify the frozen contract for zero demo gain.
- **UART intake:** no new endpoint. The UART bridge script on the laptop reads the pack and POSTs a single-battery JSON array to the existing `/batteries/ingest`. Hardware enters through the same front door as CSVs — which is itself a demo line ("same pipeline").
- **Errors:** envelope `{"error": {"code", "message"}}` (docs/04); per-battery failures inside the pipeline are logged + skipped, never fatal (docs/08).

## Non-goals (Antigravity: do not build)

User accounts, roles, rate limiting, queues/Kafka, microservices, caching layers, multi-tenancy, blockchain, file storage services. Single process + Postgres carries 847 batteries with seconds to spare (ml-plan/14 perf gate).
