# 06 — Battery Aadhaar Service Design

## Identifiers (two, with distinct jobs)

- **BAT-ID** = `batteries.external_ref` ("BAT-001") — the operator's handle, arrives in the upload, shown in tables.
- **Aadhaar ID** = 21-char BPAN-style string (docs/03 format: country·mfr·chemistry·voltage·capacity·date·serial → `INFOAN480415032400231`), generated by `services/aadhaar.issue()` after assessment, unique-indexed. Both encode and **decode** are implemented — the live segment-by-segment decode is a demo beat (docs/03).

## issue() — exactly what happens

1. Build the 21-char ID from battery metadata; collision-check via unique index (serial suffix bumps on conflict).
2. Backfill historical lifecycle events from `manufacture_date`: `manufactured`, `first_life_started` (+2–6 weeks, payload `{vehicle: "<form factor>, <city>"}`), `retired_from_ev` (recent). Then append live events: `ingested`, `assessed` (payload: soh/grade), `aadhaar_issued`.
3. Render `life_story` (template over metadata + assessment + destination — innovation #1; deterministic, no LLM).
4. Set `batteries.aadhaar_id`; status untouched (pipeline owns status).

## Passport composition (read path — NOT a table, see 02_*)

`GET .../aadhaar` assembles the docs/04 §5 payload: `decoded` (parsed from the ID), `static` (battery columns + mass estimate), `dynamic` (latest assessment + status — deliberately mirroring BPAN's static/QR vs dynamic/server split; say so in the demo), `timeline` (lifecycle_events ordered), `life_story`, `impact` (deployment row). Public route `GET /api/v1/aadhaar/{aadhaar_id}` returns the identical payload via the unique index — no auth (it's a certificate).

## QR workflow

`qr_payload` = `https://<deployed-host>/b/{aadhaar_id}` (the blueprint P5 public page), composed from a `PUBLIC_BASE_URL` env var — **must point at the live Railway deploy**, falling back to localhost for rehearsals. Pixels render client-side (frozen docs/04/07); the printed hero-card QR comes from `scripts/make_qr_card.py` run offline at H30 (doc 12 kit). Backend owns the URL truth; the frontend owns the pixels.

## Assessment & deployment history

Both are append-only tables — "history" costs zero extra design: passport timeline shows every `assessed` and `deployment_assigned` event in order, so a re-assessed battery (knee alarm, innovation #11) naturally grows a richer passport. This is the "lifecycle OS" claim made tangible.
